## Monday 2026-07-13 — ran the full stack locally, fixed error handling, and stopped CI from rebuilding AWS

### What I did

- **Brought the whole platform up locally with `docker compose`** — the four services (monolith, product-service, order-service, gateway) plus one Postgres *per* service, mirroring the production one-RDS-per-service topology. Front door is the gateway on `:8090`. Verified end-to-end: registered a user on the monolith, logged in, and read orders from order-service with that JWT through the gateway — which exercises cross-service auth (the shared `JWT_SECRET`) and gateway routing in one flow.

- **Got the Angular frontend running against the local stack** (`:4200`). It didn't work out of the box for two reasons worth knowing:
  - Dev `environment.ts` pointed `apiUrl` at the **AWS Elastic Beanstalk gateway**, not localhost — so `npm start` was talking to a dead host.
  - Pointing it straight at `localhost:8090` doesn't work either: **the gateway has no CORS config**, so the browser's preflight from `:4200` is rejected with a 403.
  - Fix: route API calls through the **Angular dev-server proxy** (`proxy.conf.json` forwards `/api/*` → `:8090`, `apiUrl: ''` makes requests same-origin). Same-origin means CORS never applies, and the backend stays untouched. `environment.prod.ts` is unchanged.

- **Fixed client errors being reported as 500s** (all three services). The catch-all `@ExceptionHandler(Exception.class)` was swallowing two things and relabelling them "Internal server error":
  - Unmapped paths (`NoResourceFoundException`) → now a real **404**.
  - Bodies Jackson can't parse (`HttpMessageNotReadableException`) → now **400**.
  - It also **discarded the stack trace**, so genuine 500s left no trace in the logs. Added logging — and it immediately exposed a second bug (below).

- **Boxed the request-DTO primitives.** `ProductRequest.price`/`stockQuantity` and `OrderItemRequest.quantity` were primitives (`double`, `int`). A missing field died during *deserialization*, before Bean Validation ran, so `@Positive` never fired and the client got an opaque 500. Boxed them to `Double`/`Integer` + `@NotNull` — a missing field now returns the proper per-field 400 (`{"price":"Price must not be null"}`).

- **Gated all six AWS deploy workflows behind `workflow_dispatch`.** This was the important one. Every deploy pipeline still fired on push to `dev`/`master`, and the monolith pipeline is *self-healing* — it **recreates the EB environment when it's missing**. With free-tier credits exhausted and the environments torn down, a routine push was an attempt to rebuild billable infrastructure. A push today got as far as ECR login before I cancelled it (no EB env created, no spend). Deploys are now manual-only from the Actions tab.

### Problems and how I resolved them

- **`/actuator/health` on the monolith returned a 500.** Not an app failure — the monolith has no actuator dependency, so the path 404s, and the catch-all handler dressed the 404 up as a 500. Symptom of the real bug above.
- **The push that triggered four AWS deploys.** Caught it mid-run and cancelled. The fix (manual-only deploys) is committed.
- **CORS: investigated, changed, then reverted.** I found CORS is configured in only two of four services (monolith + product-service), each hardcoding `localhost:4200`, with none on the gateway or order-service. That means the *deployed* CloudFront frontend can't preflight through the gateway. I centralised CORS at the gateway and deleted the per-service configs — then **reverted it all**. Reason: it changes behavior on a deployed path, nothing was actually broken for local work, and the local dev-proxy already sidesteps CORS entirely. Parked as a real finding, not an urgent fix.

### What I'm planning

- Get back to the **language-depth work** (core Java: streams, Optional, records, generics, equals/hashCode; then concurrency) — that's still the priority from Friday and I drifted off it today.
- Decide separately whether the CORS centralisation is worth doing, when the AWS environments come back.

### Key takeaways

- **A catch-all exception handler is a liability if it doesn't log.** It converted every 404 and every malformed body into a 500, *and* threw away the stack traces that would have shown why. Adding one `log.error` surfaced a second bug within minutes. If you catch everything, you must log everything.
- **Jackson deserializes before Bean Validation runs.** Validation annotations are useless on a primitive that can't hold the invalid state in the first place — a missing `double` fails at *parse* time, where `@Positive` never gets a say. Rule: **request DTOs should use boxed types**, and let `@NotNull` do the work.
- **`curl` doesn't care about CORS; browsers do.** Every backend test passed while the browser path was broken. It's a whole class of bug that server-side testing structurally cannot see.
- **CI that auto-deploys is a liability when the environments are gone.** A pipeline designed to *recreate missing infrastructure* is exactly the thing you don't want firing on a routine push when you're on a strictly-free account. Worth knowing what your pipelines do on push before you push.
- **Scope discipline.** I set out to run the project locally and ended up changing error handling, DTOs, CI, and CORS across four services. Some of it was worth it; the CORS change wasn't, and I reverted it. Finding a problem isn't the same as it being the right moment to fix it.

## Wednesday 2026-07-15 — Spring container internals, from "what is a bean" to reading the live object graph

> (Tuesday 2026-07-14 — Bastille Day, off.)

### What I did

- **Structured the interview prep around retention.** Understanding the patterns is the easy part; the skill that matters in an interview is recalling them cold and articulating them under pressure. Retrieval practice (answering cold, from memory) and concrete lived stories are what make knowledge durable. Acted on it: built **`INTERVIEW_STORIES.md`** — 15 war stories mined from this repo's actual git history (dual-write → Outbox, the Spring Cloud Weight bug → custom `GlobalFilter`, Hibernate 7 overwriting my UUIDs, Testcontainers vs H2, N+1 by query-counting, the HTTP-in-transaction anti-pattern, the Flyway custom-bean prod incident, the Strangler-Fig extractions), each written as `problem → options → decision → tradeoff → outcome`, with a mechanism section for the underlying plumbing.

- **Mastered the Spring IoC container from the ground up.** A *Spring bean* is not a JavaBean (class with getters/setters) — it's **an object Spring constructed and holds in the ApplicationContext** (a map of ~hundreds of singletons), needs no getters, and is a single shared instance for the whole app. Having that precise definition makes the rest of the framework legible.

- **Internalised the two-phase lifecycle.** **Phase 1 (startup)** builds the entire object graph once — component-scan, construct beans bottom-up (dependency injection = Spring writing the `new`/wiring code I'd otherwise hand-write in `main()`), generate proxies, run Flyway, start Tomcat. **Phase 2 (runtime)** is requests flowing through pre-built objects — *nothing is constructed per request*. "Startup builds the machine; runtime runs it." Frontend analogy that maps it cleanly: the ApplicationContext is my React providers, `new QueryClient()` is a bean, a click is an HTTP request.

- **Worked through the concurrency model.** Thread-per-request (~200 Tomcat threads), and why that forces beans to be **stateless** (one shared singleton across 200 threads → a mutable field is a race condition / cross-user data leak); concurrency vs parallelism (200 concurrent, only ~cores-many truly parallel — it works because requests spend most of their time *blocked on I/O*, not computing); and **virtual threads (Java 21)** — park a blocked virtual thread and reuse the OS thread, so blocking I/O gets cheap with plain synchronous code, while a bounded DB connection pool remains the real ceiling.

- **Proved all of it by printing the live container.** Wrote a throwaway test (extending the Testcontainers base so it boots the *real* context against real Postgres) that dumped the bean map, confirming the model against the running system: **464 beans** built at startup; `orderJpaRepository` is actually **`$Proxy195`** (a class Spring Data fabricates at runtime — I only write the interface); `OrderService` returns the **same identity hash** on two fetches (singleton, proven); and `orderService` is wrapped as **`OrderService$$SpringCGLIB$$0`** — the `@Transactional` proxy, which is *exactly* what causes the self-invocation trap (an internal `this.method()` call bypasses the wrapper). Deleted the test after.

### Problems and how I resolved them

- **Turned vocabulary into a first-class drill target.** An earlier graded revision (`REVISION.md`) showed my strength is in *why we built it* and the next step is sharpening the precise mechanics and terminology — *port, adapter, dependency inversion, proxy, idempotent*. Adjusted the method to drill *producing* the exact term, not paraphrasing it, so the language is interview-ready.
- **Grounded the abstractions in running code.** To make the container/singleton/proxy model concrete rather than theoretical, I ran it and read the actual object graph — seeing `$Proxy195` and the matching identity hashes turned the concepts into something verified end-to-end.

### What I'm planning

- Finish the **Repository Pattern drill** (was mid-question: defending the port + adapter against a "this is over-engineering, inject the Spring Data repo directly" challenge), then drill my two weakest blocks — **HTTP client** (4/20) and **Kafka** (8/20) — both already written up in the story bank.
- Space the re-drills instead of cramming: get quizzed cold on beans / lifecycle / repository next session rather than re-reading.
- Still owed from Monday: the core-Java language-depth track (streams, Optional, records, generics, equals/hashCode) and concurrency.

### Key takeaways

- **A Spring bean is an instance in the container, not a JavaBean.** The precise definition unlocks the whole mental model: a bean is an object Spring built and holds; singleton by default; needs no getters.
- **Startup builds the machine; runtime runs it.** Every object exists before the first request. DI is just Spring writing the wiring code I'd otherwise write by hand, in dependency order, bottom-up.
- **Shared singletons force stateless beans.** ~200 threads share one object, so a mutable field is a race condition — state belongs in local variables (per-thread) or the database.
- **The `@Transactional`/`@Cacheable` proxy is a real object you can see.** `OrderService$$SpringCGLIB$$0` in the bean map *is* the self-invocation trap made visible: internal `this.` calls skip the wrapper, so the annotation silently does nothing.
- **Verifying against running code makes it durable.** Printing the live bean map turned three abstract concepts into one screen of end-to-end proof — the fastest path to knowledge that stays.

## Thursday 2026-07-16 — Repository Pattern under interrogation + the cost of a network call

### What I did

- **Closed the Repository Pattern as an interview block** via a "skeptical tech lead" defense drill (*"delete the port and adapter, just inject the Spring Data repo directly — it's over-engineering"*). Built the full defense as four beats: (1) **port** in the domain, **adapter** in infrastructure (job-ad / candidate); (2) the dependency arrow points **inward** — infrastructure depends on the domain, i.e. **dependency inversion**; (3) concrete payoff #1 — `OrderServiceTest` mocks the port and runs with **no Spring context and no database**, in milliseconds; (4) concrete payoff #2 — the port is a **curated API**: `JpaRepository` exposes ~20 methods incl. dangerous bulk deletes, the port narrows it to the 7 the domain needs, so the **domain defines the contract, not the framework**. Plus the senior move: **concede** that on a small CRUD app the layer is overhead — judgment beats dogma.

- **Sharpened the "name the principle, then cash it in" habit.** Learned to never stop at *"to respect Clean Architecture"* — an interviewer reads that as book-recitation. Always follow the principle with its concrete Monday-morning consequence (here: fast, isolated unit tests).

- **Worked the HTTP Client block** — what actually changes when a local DB call (`productRepository.findById`) becomes a cross-service HTTP call (`productServiceClient.getById`). Framed it via the analogy that unlocked it: **order-service became a frontend** — it's now a *client*, with every problem a `fetch()` has. Produced the four differences cold: (1) **slower** (~1ms → ~50ms, and `create()` calls it once *per item*); (2) **new failure modes** — down, 500, 404, and the sneaky one, **silence** (a network call can hang forever; a local call cannot); (3) the dependency can be **down while I'm up** (partial failure — doesn't exist in a monolith); (4) **no rollback** — an HTTP call is outside the transaction and can't be undone. Defenses map 1:1 to react-query: **timeout** (`AbortController`), **retry** (only if idempotent), **circuit breaker** (fail fast on a dead service).

- **Started the DB-in-production block.** Drilled: *fast in dev / slow in prod* → the real causes are **data volume** (20-row preprod vs 10k prod) and **missing index → full table scan**, diagnosed with **`EXPLAIN ANALYZE`**; building an index without downtime → **`CREATE INDEX CONCURRENTLY`**; and race conditions on the last stock unit → **optimistic** (`@Version`, checks later) vs **pessimistic** (`SELECT ... FOR UPDATE`, locks first) locking, plus the cleanest fix of all: one atomic `UPDATE ... WHERE stock >= 1`.

- **Ran a live war-room demo against the real DB.** Used `docker exec` into the running `postgres-orders` container to run `pg_stat_activity` and `EXPLAIN ANALYZE` on the real `orders` table — saw the **10 idle Hikari pool connections** in the flesh, and an **`Index Scan`** (0.58ms) confirming the table is properly indexed.

### Key takeaways

- **A principle is only worth naming if you immediately cash it in.** "Dependency inversion" alone is circular; "…so my unit tests run with no database" wins the argument.
- **An HTTP call is a letter to another country** — late, damaged, or never arriving, and you may never learn which. Timeouts are non-negotiable.
- **The connection pool lives in the app, not the DB**, and `@Transactional` holds a connection for the *whole* method — which is exactly why an HTTP call inside a transaction is a time bomb.
- **Don't guess at slowness — ask the database.** `pg_stat_activity` (now), `pg_stat_statements` (over time), `EXPLAIN ANALYZE` (why): now / history / why.

## Friday 2026-07-17 — Database production war-room: locks, deploys, pools, replicas

### What I did

- **Built the war-room diagnostic model** for "100% CPU, everything slow": don't guess — `pg_stat_activity` for what's running *right now*, `pg_stat_statements` for the top offenders *over time*, `EXPLAIN ANALYZE` the worst one, and always ask **"what changed since yesterday?"** (a deploy, a new query, data crossing a size threshold). Same principle as the k6 load test: measure until the system tells you where it hurts.

- **Schema migrations vs rolling deploys** — the non-obvious one. A `RENAME COLUMN` migration succeeds and *then* production throws errors for a few minutes before recovering "on its own." Cause: a deploy is **rolling**, so there's always an overlap window where **old code runs against the new schema** — the old pods query a column that no longer exists, and the errors stop only when the last old pod dies. Rule: a migration must be **backward-compatible with the code still running**. Never rename in one step — **expand → migrate → contract** (add new column → backfill + dual-write → drop old, across releases). Corollary drilled: *adding a nullable column is safe* (old code ignores it); `NOT NULL` without a default is not.

- **Deadlocks** — two transactions each holding what the other needs (A locks product 1 then wants 2; B locks 2 then wants 1). Postgres **detects the cycle and kills one victim** (`deadlock detected`); the other proceeds and the victim retries. Prevention is the classic one-liner: **always acquire locks in the same order** (e.g. sort the IDs) — no consistent cycle, no deadlock.

- **Connection-pool exhaustion** — the diagnostic signature to memorize: **"everything hangs + database idle" = pool starvation**, not DB overload. Traced it to my own `OrderService.create()` bug: `@Transactional` borrows 1 of ~10 Hikari connections at method start and holds it through the **hanging HTTP call** (no timeout); 10 slow creates drain the pool, so even a trivial `GET` hangs waiting for a connection. Contrast nailed down: *slow + DB at 100% CPU* = bad query / missing index; *hangs + DB idle* = pool exhaustion. The DB's CPU tells you which side of the pool the problem is on.

- **Read replicas, replication lag & the WAL.** Primary handles writes, replica handles reads. A user creates an order (primary) then lists orders (replica) and doesn't see it → **replication lag** breaking **read-your-writes** consistency; replicas are only **eventually consistent**. Mechanism: the primary journals every change to the **WAL (Write-Ahead Log)** — written sequentially to disk *before* the table is touched — and **streams** those entries to the replica to replay (incremental, never a full copy). The same WAL powers three things: **crash recovery** (replay on restart), **replication** (stream to replicas), and **point-in-time backups**. Fixes for the stale-read symptom: route post-write reads to the primary, or pin that user's reads to the primary briefly — the consistency requirement depends on the screen.

### Key takeaways

- **A rolling deploy means two code versions run at once.** Every schema change has to survive the overlap — expand/contract, never a one-shot rename or drop.
- **Deadlocks need a cycle; a consistent lock order makes cycles impossible.** Postgres won't hang forever — it shoots one and lets the other live.
- **"Everything hangs + DB idle" ≠ DB problem.** The bottleneck is in front of the database — usually the connection pool, usually something holding connections during I/O.
- **The WAL is the database's redo journal** (frontend analogy: a redux action log / event sourcing) — write the intent first, replay it to recover, sync, or restore. One mechanism, three superpowers.
- **Every DB decision is a tradeoff tied to the use case.** Index = fast reads, slower writes. Replica = scale reads, lose freshness. Pessimistic lock = safe, less concurrent. "It depends on the requirement" *is* the senior answer.
