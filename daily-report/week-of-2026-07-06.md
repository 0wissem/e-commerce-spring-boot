## Thursday 2026-07-09 — MAIF technical test + started a Backstage developer portal for the platform

### What I did
- **Presentation training and technical-question preparation** — continued the run-up.
- **Technical test with MAIF.**
- **Started standing up a Backstage developer portal** for the e-commerce platform. Backstage is an internal developer portal (a "software catalog"): one place that lists every service, who owns it, how the services depend on each other, and their live API docs. I began by describing the platform to it — a catalog descriptor per service (monolith, product, order, gateway) declaring ownership, the team/domain/system they belong to, and the dependency links between them (e.g. the order service depends on product + monolith).
- Wired each service's **live OpenAPI spec** into its catalog entry, so the portal can render the Swagger docs pulled straight from each running service.

### What I'm planning
- Finish wiring the Backstage portal: get the OpenAPI specs rendering reliably and tidy the config, then commit the catalog descriptors.

### Key takeaways
- **A developer portal is a map, not code.** Its value is making the system legible — for a team lead, having ownership, dependencies, and API docs in one browsable place is how you onboard people and reason about impact without reading every repo.

## Friday 2026-07-10 — finished the Backstage portal and documented how to run it

### What I did
- **Got the portal rendering reliably.** The OpenAPI docs now load fast and correctly for every service — the last piece from yesterday. The root cause of the earlier slowness was running a *development server* (which compiles pages on demand — 25–30s each inside a limited container) instead of a *production build* (everything compiled ahead of time and served as static files on a single port). Switching to the production build dropped page loads from ~25s to ~30ms.
- **Wrote a runbook** (`BACKSTAGE.md`) so the portal isn't tribal knowledge: what's in the catalog, the exact command to build and serve it, why the production build matters, and the gotchas already solved (the container quirks, the service-name URLs for the live OpenAPI, guest login). A team lead's job includes making things reproducible for others, not just making them work once.

### What I'm planning
- Move to the language-depth part of the plan: **core Java** (streams, Optional, records, generics, equals/hashCode) and **concurrency** — the areas probed hardest in technical tests.

### Key takeaways
- **"It works on my machine" isn't done.** The portal already ran, but it only lived in my head — a short runbook is what turns a personal hack into something the team can use.
- **Dev server vs production build** is a distinction that shows up in every stack: one optimizes for editing, the other for serving. Knowing which you're running explains a whole class of "why is this so slow" confusion.
