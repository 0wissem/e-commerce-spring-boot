## Tuesday 2026-06-09 — Preprod environment + Flyway production incident

### What I did
- Designed and stood up a **preprod environment** so the upcoming order-service work can be validated before it reaches production:
  - `dev` branch → **preprod** (region eu-west-3; the three services run under a single Elastic Beanstalk application).
  - `master` branch → **prod** (region eu-north-1; separate applications — unchanged).
  - Local development stays on H2.
- Provisioned 3 preprod PostgreSQL RDS instances and 3 Elastic Beanstalk environments (monolith, product-service, gateway). Kept the container registry (ECR) shared in eu-north-1 across both environments.
- Reworked the 3 CI/CD pipelines to select environment by branch: per-environment application name, region, and secrets, plus a manual dispatch option with an environment input.
- Removed the obsolete weighted-routing filter from the gateway (product-service has been at 100% of traffic since Phase 4, so weighted routing is no longer needed).
- Implemented **per-environment database seeding** for product-service: shared schema migrations, a full 10k-product seed for prod, and a lightweight 20-product seed for preprod, selected per environment via configuration.
- Validated both environments end to end (reads and order creation) after the fixes below.

### What I'm planning
- Treat preprod as the mandatory pre-production gate for every order-service step.
- Keep preprod cheap and isolated (small seed, dedicated databases) so it can be reset and broken safely during testing.

### Problems & solutions
Standing up a new cloud environment surfaced a chain of issues, each one uncovering the next:

1. **Capacity limit in the first AWS region** → recreated the preprod environments in a second region (eu-west-3).
2. **Wrong container image deployed.** Because the three preprod services share one Elastic Beanstalk application and the pipelines tagged versions only by commit hash, the first service to deploy a given commit "claimed" that version and the others reused the wrong image. **Resolved** by making the version label unique per service.
3. **Database connection timeouts** — the RDS security group didn't allow the application instances. **Resolved** by adding the PostgreSQL inbound rule.
4. **"Database does not exist"** — the RDS instance name is not a database name; the default database is `postgres`. **Resolved** by correcting the JDBC URLs.
5. **Temporary 502s on first boot** — expected; the seed migrations insert a large dataset on a fresh database. Resolved by waiting for completion.
6. **Production outage (resolved).** A change intended to control preprod seeding/reset introduced a stricter database-migration startup path and was released to production. It crashed the production product-service because that database's schema predates the current migration tooling (its tables were created by an earlier mechanism and have no migration-history record), which the stricter path refuses to start against. **Resolved** by rolling production back to the previous version immediately to restore service, then reverting to the tolerant (standard) startup path while keeping the per-environment seeding. Production was redeployed and verified healthy with no data loss.
7. **Preprod order creation failing.** The preprod monolith was still pointing at the production product-service, so preprod product IDs weren't found. **Resolved** by repointing it to the preprod product-service; verified an order creates correctly with its product snapshot.

### Key takeaways
- Don't change the production database-startup path lightly — production's schema history differs from the newer environments, so only the standard, tolerant configuration is safe there.
- Every merge to `master` is a production release and must be treated as one.
- Validate the write path (order creation), not only reads — the preprod misconfiguration only showed up on order creation.

### End-of-day status
- **Production:** all read and write flows healthy (10,019 products, 8 categories, 50,003 customers, order creation OK). One known pre-existing issue: product full-text search returns an error in production — unrelated to this week's work, but worth investigating since it's the endpoint that originally justified extracting product-service.
- **Preprod:** fully functional end to end (20 products; order creation with snapshot verified).

  - Rewrote both `application-dev.properties` files to point the `dev` profile at a local Postgres container (env-overridable URLs: monolith → `monolith` DB, order-service → `orders` DB), relying on the base config's PostgreSQL dialect instead of overriding to H2.
- **Initialised the order-service microservice** — the next Strangler Fig extraction after product-service:
  - Rewrote the extraction plan (`ORDER_SERVICE_PLAN.md`) as a **big-bang cutover strategy** rather than a weighted/dual-write migration.
  - Scaffolded the empty service shell: Maven wrapper, `pom.xml`, `Dockerfile`, base + dev properties, and a placeholder `V1__create_orders_tables.sql` Flyway migration documenting the `orders` / `order_items` data model (customer_id and product_id as cross-service references with NO foreign keys; product_snapshot as JSON-in-TEXT — the same JSONB snapshot pattern proven on product-service).
- **Preparing the local development environment** around the new Postgres-in-Docker setup so the services can be run and exercised locally against the same engine as the cloud.

### What I'm planning
- Build out order-service layer by layer (domain → application → infrastructure → api), following the product-service playbook and the big-bang cutover plan.
- Fill in the real `V1` DDL for orders / order_items and wire up the local Postgres database for the new service.

### Problems & solutions
- **IntelliJ environment broken on macOS** → repaired the IDE setup before any code work; no impact on the codebase.
- **Two databases in local dev (H2) vs cloud (Postgres) was a hidden risk** — dialect and SQL differences only surface in the cloud. **Resolved** by removing H2 entirely so every environment runs PostgreSQL; the only difference between dev and prod is now the connection target.

### Key takeaways
- Local should match production as closely as possible — running Postgres in Docker locally removes a whole class of "passes on H2, fails on Postgres" surprises (and is consistent with the V1 migration note about keeping types portable, now moot since there's only one engine).
- Scaffold first, implement second: the order-service shell + plan are in place before writing any domain logic, so the extraction follows a deliberate playbook rather than ad-hoc growth.

## Thursday 2026-06-11 — order-service domain layer + local cross-service testing

### What I did
- **Validated the new local PostgreSQL setup end to end** by running the existing services on my machine and exercising their full APIs against local databases:
  - Ran the monolith (customers + orders) and product-service (products) locally, each against its own local Postgres database, and tested every endpoint — list / read / create / update / delete on customers, orders, and products, plus input validation.
  - Confirmed **cross-service communication works locally**: creating an order in the monolith calls product-service over HTTP to fetch the product and freeze its name, price, and categories into the order's snapshot. Verified both the success path and the failure path (ordering a non-existent product correctly fails, because the lookup to product-service returns "not found").
- **Built the order-service domain layer** — the first real code of the new service:
  - `Order` and `OrderItem` entities with a proper parent/child relationship (an order owns its items; removing an order removes them).
  - An `OrderStatus` lifecycle (PENDING → CONFIRMED → SHIPPED → DELIVERED / CANCELLED).
  - The product snapshot as a value object stored as JSON text via a dedicated converter — the same "freeze the product at purchase time" pattern proven on the monolith, so an order keeps the price and details it was placed with even if the product changes later.
- **Got order-service compiling and running locally** against its own `orders` Postgres database, with the schema generated correctly (parent/child foreign key, snapshot stored as text, statuses stored as readable strings).
- **Hardened the data model while the tables are still empty** (cheap to get right now, expensive to migrate later):
  - Store both a customer **reference** (`customer_id`) and a customer **name snapshot** — the reference links back to the customer; the snapshot avoids a lookup on every read, since order-service has no customers table of its own.
  - Money stored as a decimal, not an integer (the integer version was silently dropping the cents).
  - Database-generated UUID identifiers.
  - Order status stored as readable text with a database check constraint, instead of fragile numeric codes that would shift meaning if the list is ever reordered.
- **Committed and pushed** the domain entities, the snapshot converter, and the service entry point.

### What I'm planning
- Build the layers above the domain — repository, application/service, and the REST API — plus the HTTP clients order-service needs (to product-service for product data, to the monolith for the customer name).
- Write the real `V1` database migration (orders / order_items DDL), then rehearse the data migration and cutover on preprod before touching production, per the big-bang cutover plan.

### Problems & solutions
- **product-service doesn't seed when run locally.** Its database migrations (and therefore the sample-data seed) don't run on a fresh local database — the schema gets created by the ORM's fallback instead, and no data is loaded. This is the **same underlying cause as Tuesday's production incident**: the service has no migration-history record, so its migration tooling doesn't engage. For local testing I worked around it by creating a product through the API; flagging it as a known issue to fix properly, since it now affects both production and local dev.
- **Persisting the product snapshot needed an explicit converter.** The snapshot is a structured value and the ORM can't store it in a column on its own. Resolved by adding a converter that serialises it to JSON text on save and parses it back on read — and forcing the column to a `text` type so larger snapshots aren't truncated.

### Key takeaways
- Get enum-as-text and money-as-decimal right from the very first migration — these are almost free to fix on an empty table and painful once there's data.
- Running the same database engine locally as in the cloud paid off immediately: the order-service schema, the foreign key, and the snapshot column all behaved identically to what production will see.

## Friday 2026-06-12 — order-service complete vertical slice + migration cleanup

### What I did
- **Finished the order-service implementation — all four layers now run end to end:**
  - *Application:* the order service itself (the create-order use case calls the monolith for the customer name and product-service for each product, then freezes both as snapshots), plus the service interface, the mapper, and the request/response DTOs.
  - *Infrastructure:* the repository (port + adapter + data-access interface) and two HTTP clients — one to product-service, one to the monolith.
  - *API:* the REST controller (`/api/orders`) and a global exception handler that turns "not found" into 404 and validation failures into 400.
- **Verified the whole slice end to end** against local Postgres, with the monolith and product-service running: created an order through order-service and watched a single request fan out to both other services, snapshot their data, and persist it; confirmed pagination, validation (400), not-found (404), status update, and delete all behave correctly. The response JSON matches the monolith's exactly (same field names), so the eventual switchover stays invisible to the frontend.
- **Two corrections surfaced while wiring it up** (details below): a field-naming bug that blocked startup, and a folder reorganisation to match the other services' layout.
- **Committed and pushed** the complete service.
- **Cleaned up the database migrations across all three services** — the big win of the day:
  - **Found the root cause of our long-running migration trouble: one missing dependency.** Spring Boot 4 split its auto-configuration into per-feature modules, and the module that actually *runs* database migrations on startup was never added to any service. So migrations silently never ran, and the schema was being built by the ORM's fallback instead. This single gap explains the whole history: the monolith only worked via a hand-written workaround, product-service broke production when that workaround was removed, and the "fresh sample rows on every redeploy" annoyance.
  - **Added the missing module to all three services**, then **collapsed each service's messy migration history into one clean "baseline"** containing structure only — no sample data. Removed the seed files entirely (test data now comes through the APIs), and set every service to **validate** its schema against the code at startup, so migrations are the single source of truth.
  - **Fixed the production search bug as a side effect.** Product full-text search has been failing in production for weeks; the cause was the same one — the search column lived in a migration that never ran in production, so the column simply didn't exist there. The new baseline includes it, and search now works end to end.
  - **Verified all three on fresh databases:** migrations apply once, validation passes, search works, and the monolith now starts in ~2 seconds instead of ~75 (no more replaying 22 migrations and seeding hundreds of thousands of rows). Committed and pushed.

### What I'm planning
- **Rebuild the production and preprod databases empty** so the new baselines can apply cleanly (the existing databases have no migration history), remove the now-obsolete seed configuration, then deploy the cleanup — this is also what finally ships the production search fix.
- Resume the order-service cutover track: CI/CD pipeline, AWS environment, gateway route, and the preprod cutover rehearsal before production.

### Problems & solutions
- **A startup failure caused by field naming.** The repository's "find orders by customer" query is generated automatically from the entity's field name. Our fields used database-style naming (`customer_id`), so the expected property couldn't be found and the service refused to start. Resolved by switching the entity fields to standard camelCase (`customerId`) — the database column names stay the same, so there's no schema change.
- **Folder structure didn't match the package names**, which had been causing IDE warnings since the service was created. Reorganised the files so folders and packages line up, matching the layout already used by product-service. Pure file moves — no behaviour change.

### Key takeaways
- **Migrations are for schema, not data — this is the root of our seeding mess.** The annoyance of getting fresh sample rows on every redeploy comes from sample data living *inside* the migration pipeline. The fix is to take all sample data out of migrations entirely: migrations build structure and run exactly once; data comes from the application; production is never seeded. This also makes future migrations trivial to test (empty database → run migrations → done, with no seed noise).
- A microservice only proves itself once the **whole vertical slice** runs together: the order-service create path is the first time we've seen it own its data, call two other services, and snapshot their responses — all behind a single HTTP call.
- **A whole category of recurring problems traced back to a single missing dependency.** Three separate symptoms over two weeks — a startup workaround, a production crash, and a broken search endpoint — were all the same root cause. Worth the discipline of finding the *one* underlying reason rather than patching each symptom: fixing it once cleared all three and made future migrations boringly predictable.
