### What I did
- **Completed the test suites for the two remaining services.** Last week I built the full suite for the product service; I extended the same approach to the **order service** and the **monolith** — unit tests (mocked dependencies), web-layer tests (simulated HTTP requests checking status codes, validation, and error shapes), and a context/integration smoke test against a real database. The backend now has **48 automated tests across the 3 services, all green.**
- **Added code-coverage measurement (JaCoCo) to all three services** so I can see, per class, exactly what the tests exercise — product ~64%, order ~65%, monolith ~69%.
- **Closed a real coverage gap:** the product service's "category" area was almost untested (~10%); I added its unit + web tests, lifting the service's *useful* coverage by ~15 points — its business logic is now near-fully covered.
- **Added a CI gate:** a pipeline that automatically runs **every test on each push / pull request**. A failing test now turns the check red before anything can be merged — the last missing piece versus a professional setup.
- **Explored adding real server-side security** to the product service with Spring Security (today the backend is open — auth is only enforced on the frontend). I stood up a working version (public reads / authenticated writes) with a test proving an unauthenticated write is rejected *before* it ever reaches the controller — security as a cross-cutting layer, not code in the controllers. Then I **chose to redo it properly as JWT** (matching the token the frontend already uses, and the far more real-world approach) and reverted the quick version cleanly — nothing had been committed, so the codebase stays at its solid, all-green state.

### What I'm planning
- Implement **JWT** properly: a login endpoint that issues a signed token + token validation on every request (resource server), with tests covering login → token → authenticated call.
- Continue the backend study plan after that: caching, then core-Java and concurrency refreshers — using the project as the lab.

### Key takeaways
- **Coverage is a signal, not a goal.** 100% on trivial data classes is meaningless; the value is covering business logic. Filling the untested "category" area raised *useful* coverage far more than adding token tests on getters would.
- **Judgment over speed.** A Basic-auth shortcut would have "worked" but demonstrated the wrong thing; doing it as JWT is more valuable for both the project and interview preparation. Knowing when *not* to keep a working-but-wrong solution is part of the job.

## Tuesday 2026-06-30 — turned the monolith's "customers" into a users/identity model and added JWT authentication

### What I did
- **Reworked the data model into a proper identity table.** The monolith's `customers` table became `users`, gaining a **role** (`ADMIN` / `CONSUMER`) and a **hashed password** — done as a versioned DB migration, so existing rows are preserved and become consumers. This turns the monolith into the platform's identity owner (the natural place, since it already owned customers).
- **Implemented JWT authentication (the real, token-based approach the frontend already uses).** Added a **login / register** endpoint that verifies credentials (passwords hashed with BCrypt) and issues a **signed token** carrying the user's role. Every incoming request is then validated against that token, statelessly (no server-side session).
- **Kept read access to customers open on purpose**, so the order service can still fetch a customer's name when creating an order — i.e. I secured the write operations without breaking the existing cross-service contract.
- **Covered it with tests** (register → token, login with right/wrong password → 200/401, protected write rejected without a token).

### What I'm planning
- Add **role-based authorization** on top (admin-only operations) and then verify the whole thing runs end-to-end across services.

### Key takeaways
- **Authentication = "who are you"** (the token), separate from **authorization = "what may you do"** (the role) — two distinct layers, done in order.
- Securing a shared service means watching the **contracts other services depend on**: locking everything down would have broken order creation, so only writes were protected.

## Wednesday 2026-07-01 — added role-based authorization and proved the whole change works end-to-end across services

### What I did
- **Added role-based authorization.** Sensitive operations (e.g. deleting a user) are now **restricted to admins** declaratively; a normal consumer calling them is cleanly rejected with a "forbidden" response, while an admin succeeds. Verified with tests for both roles.
- **Validated the whole change end-to-end on the running stack**, not just in unit tests: I rebuilt and restarted the monolith with the new code, then **created a real order through the order service** — which calls the renamed, now-secured monolith to fetch the customer name. The order was created successfully with the correct customer name, confirming **nothing broke** for the other services.
- Net result: the monolith is now a real **identity provider** (users + roles + JWT), and the migration + auth were introduced **without disrupting** the order or product services.

### What I'm planning
- Continue the backend plan: caching, then core-Java and concurrency refreshers.
- Optionally extend the same JWT validation to the product and order services later (they're still open today).

### Key takeaways
- **Unit tests prove a piece; an end-to-end run proves the system.** Rebuilding the stack and placing a real order gave the confidence that a refactor of this size (renaming a core table + adding security) didn't break the contracts between services.
- **Introduce security incrementally.** Doing the monolith first, end-to-end, is safer and clearer than trying to secure every service at once.

## Thursday 2026-07-02 — extended JWT validation to the other two services and verified the whole platform end-to-end

### What I did
- **Rolled out the JWT security across the whole backend.** Yesterday only the monolith enforced authentication; today the **product** and **order** services also **validate the same token** (they share one signing secret with the monolith, so a token issued at login is accepted everywhere).
  - **Product service:** browsing the catalogue stays public (so the order service can still read product data without a token), but **catalogue changes now require an admin**.
  - **Order service:** every order operation now **requires a valid token** — you must be logged in to place or view orders.
- **Kept the services talking to each other.** The internal service-to-service calls use the public read endpoints, so tightening security didn't break order creation.
- **Verified the whole thing end-to-end on the running stack** (not just tests): an order placed **without a token is rejected**, with a **valid token it succeeds** and correctly pulls the customer name from the monolith and the product details from the product service. Also confirmed the role rule live — a normal user trying to modify the catalogue is refused.
- **Made the shared secret explicit in the container orchestration** (docker-compose), overridable per environment instead of relying on a built-in default — the standard "configuration comes from the environment" practice.
- Added security tests to both services (protected calls rejected without a token, admin-only rules enforced).

### What I'm planning
- Move on in the backend plan: caching, then core-Java and concurrency refreshers.
- (Later, optional) a login button on the frontend so the stored token is actually sent on protected calls.

### Key takeaways
- **A shared secret = single sign-on across services.** Because all services trust the same key, one login works everywhere and no service needs to call back to the monolith to check a token — validation is local and stateless.
- **Security has to respect existing contracts.** The trick was protecting the *writes* while leaving the *reads* the other services depend on open — lock down too much and the platform stops working.
