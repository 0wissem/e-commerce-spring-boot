## Monday 2026-06-15 — finished the order-service extraction + monolith cleanup + containerised the stack

### What I did
- **Completed the order-service extraction, end to end (locally).** This is the milestone the last two weeks were building toward:
  - Pointed the **gateway** to send order traffic to the new order-service (everything else still goes to the monolith), so the switch is invisible to any client.
  - **Migrated the existing orders** out of the monolith and into order-service — joining the customers table on the way to rebuild the customer-name snapshot order-service needs — and verified the orders come back correctly through the gateway.
  - **Removed orders from the monolith entirely** (code and database tables). The monolith now owns **only customers**.
- **Cleaned the monolith down to a genuinely single-domain service.** Removed leftover order code, an unused configuration property, and a **dead low-stock-alert feature**: it relied on messaging but could no longer work after products were extracted (the product service emits nothing, and its config still pointed at a deleted package). The monolith no longer depends on Kafka or mail, and now boots in ~2 seconds with no extra setup.
- **Ran a full API test sweep and caught a real, long-standing bug.** I tested every endpoint across all services through the gateway; this surfaced that **creating a category via the API had never actually worked** (it tried to insert a row with no identifier → server error), hidden until now because categories came from seed data rather than the API. One-line fix; every endpoint is now green.
- **Containerised the whole stack with Docker Compose.** A single command now brings up the databases plus all four services, wired together — replacing the fiddly "start four services by hand with environment variables" routine I'd been living with. Verified the full API surface against the containerised stack.
- **Gave each service its own database.** Split the single shared database server into **one per service**, so a database failure is now isolated to a single service instead of taking the whole platform down — matching the production topology (one managed database per service). Verified that each database holds only its own service's tables.

### What I'm planning
- Move the stack onto a **local Kubernetes cluster** as the next learning step. The Docker Compose setup translates cleanly — one stateful database per service, plus a deployment and an internal address per service — so I started mapping that out. Two open choices to settle: how external traffic enters the cluster (a simple tunnel vs. a proper ingress) and how the deployment files are written (by hand vs. a templating tool).
- The cloud deployment remains paused (free-tier credits exhausted); all of the above is being done and verified locally.

### Problems & solutions
- **A category-creation bug hidden for weeks.** Creating a category always failed because the code never generated an identifier, but nobody noticed since categories were loaded from seed data, not created through the API. A full, endpoint-by-endpoint test sweep surfaced it immediately; fixed by generating the identifier the same way the product code already does.
- **Docker images wouldn't build on my Mac.** The lightweight base images we used aren't published for Apple-Silicon chips, so the build failed. Switched to the multi-architecture variants of the same images, which build on both my machine and the cloud pipeline.
- **A leftover container blocked the database restart.** Renaming a service left an orphaned container still holding a network port; cleared it with the orphan-removal option and the stack came up clean.

### Key takeaways
- **Comprehensive testing pays for itself.** One full sweep through the gateway found a real bug that had been silently broken for weeks — exactly the kind of safety net a team should have as standard, not as an afterthought.
- **Finishing an extraction means cleaning up after it.** The monolith only became genuinely simple once the order code, the dead messaging feature, and the stray config were *all* removed. The job isn't done at "it works" — it's done when the old home is clean.
- **Local should mirror production.** Giving each service its own database removed a hidden inconsistency (a shared point of failure) and, as a bonus, makes the upcoming Kubernetes work a near-mechanical translation rather than a redesign.

## Tuesday 2026-06-16 — ran the whole platform on Kubernetes, moved the databases out of the cluster, and started on observability

### What I did
- **Deployed the entire stack to a local Kubernetes cluster.** All four services plus their three databases now run on Kubernetes (minikube): each service has a deployment, an internal address, and its own database; each app waits for its database before starting and reports its own health so the platform can tell when something is unwell. I verified the full API surface through the gateway — every endpoint green.
- **Watched Kubernetes scale and self-heal, for real.** I scaled the order service from one copy to four and saw the internal address automatically start sharing traffic across all four; then I deliberately deleted a running copy and watched Kubernetes recreate a replacement within seconds and route around the gap with no downtime. This is the core promise of the platform — you declare the desired state, and it continuously makes reality match — seen first-hand rather than read about.
- **Moved every database *out* of the cluster — the real production shape.** Kubernetes is for stateless applications; the data belongs in a managed database outside it (in production, a managed cloud database). I migrated all three databases out of the cluster to external instances, copying the existing data across first so nothing was lost, and pointed each service at its external database **without changing a single line of application config** — only what the database *name* resolves to. Verified the full API sweep again, including the cross-service order creation, all still green against the externalised databases. Captured the whole thing as diagrams, a reference document, and a reusable config file.
- **Wrote down a Kubernetes improvement backlog** — the path from "it runs" to "production-grade" (resource limits, automatic scaling under load, a proper front door, secret handling, and more) — so the next steps are queued and explained rather than living in my head.
- **Committed and merged all the Kubernetes work**, then **explored the cluster visually** with the built-in dashboard and re-ran the scale / self-heal demo through the UI.
- **Started on observability (metrics dashboards).** Installed the standard Prometheus + Grafana stack to get historical graphs of how the services behave over time (CPU, memory, request rates) — the layer the cluster browsers can't show.

### What I'm planning
- **Right-size the local cluster, then do observability properly.** The metrics stack is heavy and the local cluster was running on minimal resources, so this needs a clean, larger environment rather than being squeezed onto an already-full one (see Problems below).
- Continue down the Kubernetes improvement backlog: automatic scaling under load is the natural sequel to today's manual scaling demo.
- The cloud deployment remains paused (free-tier credits exhausted); everything continues locally.

### Problems & solutions
- **The metrics server wouldn't stay up.** Kubernetes' built-in resource-usage reporter kept crash-restarting (a known quirk on this kind of local setup). It only affects on-demand usage readings and automatic scaling, not the dashboards, so I noted it for the scaling work and moved on.
- **Prometheus fell into a crash loop — and the root cause was capacity, not configuration.** It would start, run for about a minute, then get killed and restart, over and over. Digging in: it was being killed for failing its own health check, because the local cluster had only been given **2 CPUs and 4 GB of memory** (the default) while already running seven application copies and three databases — memory was sitting at **97%**. Starved of resources, Prometheus couldn't answer its health check in time, so the platform kept restarting it. Not a bug — a capacity-planning lesson. The fix is to give the cluster more headroom (a planned restart) rather than to keep squeezing; for now I stopped the extra load to halt the crash loop.

### Key takeaways
- **An application shouldn't know or care where its database lives.** Because every service connects to its database by a stable *name* rather than a fixed address, moving the databases out of the cluster was a configuration change at the platform level, with zero application changes. That same indirection is what lets a managed database fail over to a standby in production without the app noticing.
- **"It runs" and "it's production-ready" are different milestones.** Getting the platform onto Kubernetes was the easy half; the backlog I wrote (resource limits, scaling, secrets, a real front door) is the half that separates a demo from something you'd trust in production.
- **Health checks need enough resources to pass.** The Prometheus crash loop was a sharp reminder that liveness checks assume the process can respond — starve it, and the very mechanism meant to keep things healthy will instead restart a perfectly good process to death. Capacity is a first-class concern, not an afterthought.

## Wednesday 2026-06-17 — gave the cluster room to breathe, then made it scale automatically under load

### What I did
- **Right-sized the local cluster.** Following yesterday's capacity lesson, I rebuilt the cluster with roughly three times the CPU and more memory (from the bare-minimum default to 6 CPUs / 6.5 GB). With the headroom in place, the crash loops from yesterday stopped and there was finally room to run the platform *and* the monitoring stack *and* extra copies of a service at the same time.
- **Set explicit resource budgets on every service.** Each of the four services now declares both what it needs to start (a guaranteed reservation) and a ceiling it can't exceed. This is what lets the cluster pack services sensibly onto the available hardware and stops any one misbehaving service from starving the others — and it's the prerequisite for automatic scaling, which can only make decisions if it knows what "normal" usage looks like.
- **Turned on automatic scaling for the product service.** I added a rule that watches the product service's CPU usage and adds or removes copies on its own to keep usage around a target — between a floor of 2 copies (so there's always redundancy) and a ceiling of 6. Yesterday I scaled by hand; now the platform does it without me, and won't scale to zero or run away unbounded.
- **Wrote it all down.** Captured the resource budgets, the scaling rule, and the reasoning behind the chosen numbers in a reference document so the decisions are explained, not just configured.
- **Committed and merged the work** onto the main development branch.

### What I'm planning
- Continue down the Kubernetes improvement backlog: a proper front door (ingress) for external traffic, real secret handling, and network rules between services.
- Return to observability now that the cluster has the headroom for it — historical dashboards on top of the automatic scaling, so I can *see* the scaling decisions happening.
- Cloud deployment remains paused (free-tier credits exhausted); everything continues locally.

### Problems & solutions
- **Changing the cluster's resources isn't a simple restart.** On this kind of local setup, the cluster's CPU and memory limits are fixed when it's first created — a plain stop-and-start won't change them. I had to tear the cluster down and recreate it at the new size, which meant reloading the application images afterwards; the external databases sit outside the cluster, so the data was untouched throughout.
- **The host machine has its own ceiling.** I couldn't hand the cluster as much memory as I first wanted, because the underlying virtualisation layer on my Mac is itself capped below the machine's total. A reminder that on a laptop the cluster is always a guest negotiating for a slice of a finite host — another reason production runs this on dedicated hardware.

### Key takeaways
- **Automatic scaling is only as good as the budgets underneath it.** The scaling rule can't make sensible decisions until each service has declared what it needs and what its ceiling is — the limits I set aren't busywork, they're the inputs the autoscaler reads. The two pieces are one feature, not two.
- **Capacity decisions cascade.** Yesterday's crash loop, today's rebuild, and the automatic scaling all trace back to the same question — how much does this need, and how much is available — at three different layers: the application, the cluster, and the host. Getting it right at one layer exposes the next.
