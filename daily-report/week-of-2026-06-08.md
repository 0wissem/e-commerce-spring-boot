# Weekly Report - Week of 2026-06-08

## 2026-06-09 (Tuesday)

### What I did

- **Set up preprod environment across all three CI/CD pipelines.** The `dev` branch now deploys to a dedicated `preprod` Elastic Beanstalk application in `eu-west-3`, while `master` continues to deploy to prod in `eu-north-1`. Added `workflow_dispatch` triggers to all three pipelines (monolith, product-service, gateway) so I can kick off preprod deploys manually without a push.

- **Fixed cross-service image collisions in the shared preprod EB app.** All three services share a single Elastic Beanstalk application for preprod. The version labels were not namespaced per service, so one service's image would overwrite another's in the EB version list. Prefixed each version label with the service name to make them unique.

- **Removed `WeightedRoutingFilter` from the gateway.** Product-service has been handling 100% of `/api/products/**` and `/api/categories/**` traffic since Phase 4. The weighted routing filter existed only for the gradual traffic split phase and was dead code. Deleted the class and simplified the gateway pipeline accordingly.

- **Split product-service Flyway seed by environment.** Schema migrations stay in `db/migration` (run in all environments). The V6 seed is now split: `db/seed-prod` keeps the full 10k-product dataset (checksum preserved, so Flyway won't re-run it on prod), and a new `db/seed-preprod` seeds 20 products for lightweight preprod testing. The active locations are controlled via the `SPRING_FLYWAY_LOCATIONS` environment variable.

- **Added self-healing preprod environment creation to the product-service pipeline.** If the preprod EB environment doesn't exist, the pipeline now auto-creates it — mirroring the self-healing logic already in place for prod.

- **Reverted the custom Flyway bean (prod hotfix).** After deploying the per-env seed split, prod product-service failed to start. The custom `FlywayConfig` bean called `migrate()` in strict mode, but the prod DB was originally bootstrapped by Hibernate `ddl-auto` before Flyway was introduced, so it has no `flyway_schema_history` table. Spring Boot's auto-configured Flyway is tolerant of this (baseline-on-migrate semantics); the custom bean was not. Deleted `FlywayConfig.java` and moved entirely to `SPRING_FLYWAY_LOCATIONS` env var to control seed location — no custom bean needed.

### Problems & solutions

- **Prod product-service failed to start after Flyway seed split.** Root cause: custom `FlywayConfig` bean ran `migrate()` strictly on a DB with no `flyway_schema_history` table (prod was bootstrapped by Hibernate before Flyway was introduced). Fix: deleted the custom bean entirely; per-env seed path now controlled via `SPRING_FLYWAY_LOCATIONS`, relying on Spring Boot's auto-configured Flyway which handles a missing history table gracefully.

- **Cross-service image collision in shared preprod EB app.** All three services were generating version labels using the same pattern (git SHA only), so whichever service deployed last would clobber the others' registered versions in the shared EB application. Fix: prefixed each version label with the service name (e.g., `monolith-<sha>`, `product-service-<sha>`, `gateway-<sha>`).

- **ECR region flip-flopped during pipeline work.** I initially split ECR by region (eu-west-3 for preprod, eu-north-1 for prod) to mirror the EB region split, then reverted — all images now push to the single eu-north-1 ECR registry regardless of environment, and only the EB deployment target switches region per branch. Simpler to operate with one registry.
