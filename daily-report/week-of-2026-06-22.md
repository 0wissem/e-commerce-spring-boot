## Monday 2026-06-22 — brought the local platform under GitOps + IaC, trialled a cloud deploy on free credits, and tore it down for zero cost

### What I did
- **Put the monitoring stack under infrastructure-as-code.** The Prometheus/Grafana stack I'd installed by hand last week is now **declared in Terraform** (as a Helm release) and applied against the local cluster. I proved the value by tearing it down and rebuilding it from code in about a minute — the same reproducibility discipline as cloud infrastructure, but entirely free and local.
- **Added GitOps with ArgoCD.** Installed ArgoCD on the local cluster and pointed it at this repository: it now **reads the desired state from Git and reconciles the cluster to match it**. I connected the application, watched it detect drift between the live cluster and the repo, and synced it back to the source of truth. This is the "the repository is the single source of truth, the platform continuously matches it" model that mature teams run.
- **Hit and fixed two real GitOps problems** (see below) — a Kubernetes immutability error and the classic autoscaler-vs-Git conflict — both resolved the canonical way.
- **Trialled the full cloud deployment on managed Kubernetes (AWS EKS), as code, using free promotional credits.** I'd previously concluded the cloud cluster can't be truly free; this time I deployed it deliberately on **free-tier-eligible machines** with **tight spending guardrails** (a low budget alert well under the credit balance), brought up the entire infrastructure from Terraform (~60 resources), and repopulated the image registry through the pipeline.
- **Validated the APIs and debugged a real failure** (see below).
- **Tore the whole cloud environment down the same session** to guarantee zero overnight spend — the disposable-environment discipline the credit budget depends on.

### What I'm planning
- Keep the **runtime learning on the free local cluster** (it already has monitoring, autoscaling, and now GitOps) and treat the cloud cluster as a **short, disposable, credit-funded exercise** — spin up, verify, tear down, never leave running.
- Next free improvements: a proper test stage in the pipelines (the biggest gap vs a professional setup), and the "app-of-apps" GitOps pattern so even the deployment definitions are managed from Git.

### Problems & solutions
- **A Kubernetes immutability error on first sync.** GitOps refused to update two database services because a core network field can't be changed in place once set (the live services pre-dated the desired config). Fixed by letting the tool delete and recreate them rather than patch — the standard way around an immutable field.
- **The autoscaler fighting Git.** One service is scaled automatically (minimum two copies), but the repository hardcoded "one copy" — so GitOps saw permanent drift and, with auto-healing, would have fought the autoscaler forever. Fixed the canonical way: removed the hardcoded count from the manifest and told GitOps to ignore that field, so the autoscaler owns it cleanly at any scale.
- **APIs returning server errors after a database change.** Two services threw 500s. Tracing the logs showed the cause precisely: the databases had been replaced underneath long-running application pods, so the apps were querying tables that no longer existed (their schema-setup step only runs at startup, and they hadn't restarted). A rolling restart let each app rebuild its schema against the fresh database, and every endpoint returned to healthy. The lesson: when you swap a stateful backend beneath a running app, the app must reconnect and re-migrate — ideally enforced by a startup ordering step, not left to chance.

### Key takeaways
- **"The repository is the source of truth" becomes real with GitOps.** Watching ArgoCD detect drift and reconcile the cluster back to Git made the idea concrete — and the two failures I hit (immutability, autoscaler conflict) are exactly the practical gotchas a team needs to know, not theory.
- **Cost discipline is a feature you build in, not a hope.** Free-tier machines, a low budget alert, a manual-only "create" action, and tearing down every session — together those make a credit-funded cloud experiment safe. The teardown is as much a part of the workflow as the deploy.
- **Read the exact error.** "Relation does not exist," with pod-start timestamps from days earlier, was the entire diagnosis of the 500s — the fix followed immediately. Precise reading beats guessing every time.

## Tuesday 2026-06-23 — got the entire platform live on cloud Kubernetes (free tier) by moving the databases out, then tore it down

### What I did
- **Got all four services running on managed Kubernetes in the cloud (AWS EKS), every API live, on free-tier machines — deployed end to end from CI.** This is the milestone I'd repeatedly concluded was impossible on free tier; it took an architecture change to crack it.
- **The breakthrough was a design decision, not a setting:** I moved the **databases out of the cluster** onto **managed databases — one per service** (the project's per-service-database design, kept intact). That freed exactly the cluster capacity the application services needed to fit on the tiny free machines.
- **Built a CI pipeline that deploys the app to the cloud cluster automatically** (keyless auth, no manual console step) — a genuine continuous-delivery step on top of the existing build pipeline.
- **Verified every endpoint** (customers, products, orders, and through the front-door gateway) returning healthy from its own managed database.
- **Then tore the entire cloud environment down to zero** — cluster, databases, network — to stop all spend. It's all infrastructure-as-code, so it rebuilds on command.

### What I'm planning
- Keep **monitoring and autoscaling on the free local cluster** — they need spare capacity the free cloud machines simply don't have. The cloud side proves the *deploy*; the local side is where I *observe and scale*.
- Treat the cloud as **disposable**: stand it up from code, verify, tear it down. Never leave it running.

### Problems & solutions
- **The microservices wouldn't all fit on the smallest (free) machines — a stack of three hard limits.** (1) The new account may only run a small number of machines; (2) each machine has a hard cap on how many application copies it can host; (3) the last service couldn't get an internal network address. I worked through each: **moved the databases off the cluster** (freeing capacity), **reduced a system component to a single copy** (freeing a slot), and **enabled a networking mode that hands out far more addresses per machine** — which finally let the last service get an address and start.
- **A deployment-tool locking clash.** Two automated infrastructure runs tried to update the shared state record at the same time and one was refused — the lock doing its job. Fixed by sequencing them so only one runs at a time.
- **A machine-group sizing rule.** You can't raise a group's minimum size above its current size in one step; I decoupled the two settings so the change applied cleanly.

### Key takeaways
- **Reach for the design change, not just more tuning.** Forcing a seven-piece system onto machines too small for it failed every which way. The thing that actually unlocked it — moving the databases out — was also the architecturally *correct* move. When tuning keeps hitting walls, the answer is often a design change.
- **Free tier has hard, knowable ceilings.** Machines-per-account, copies-per-machine, addresses-per-machine — hitting each one and understanding exactly where it sits is itself the lesson; these are the real limits a team does capacity planning around.
- **Split "deploy" from "observe."** The free cloud had no room for monitoring, so deployment lives in the cloud and observability/scaling stay local. Knowing what each environment is *for* is how I learn the whole picture without spending.
- **Proven, then destroyed.** The win isn't a running cluster — it's the **repeatable, version-controlled recipe** that stands the whole thing up on command. Keeping it running would only burn credit; the value is banked in code.

## Friday 2026-06-26 — built a real automated test suite and tackled the classic backend pitfalls (N+1, transactions)

### What I did
- **Refocused on core backend (Java/Spring) for technical-test prep** and closed the project's biggest gap: it had **zero automated tests**.
- **Built a full test suite for the product service across the testing pyramid:** fast unit tests (mocked dependencies), web-layer tests (simulated HTTP requests checking status codes + validation), and **integration tests against a real PostgreSQL** spun up in a container — including the full-text search, which can only be tested against the real database engine, not a substitute. **14 tests, all green.**
- **Demonstrated *and* fixed the "N+1 query" problem** — the most common database performance pitfall — by **measuring the actual number of SQL queries**: the naive code fired one query per item; the fix collapses it to a single query. Proven by a test, not just theory.
- **Proved transaction atomicity** on the order-creation flow: when a step fails mid-way, the whole operation **rolls back** and nothing partial is saved (verified against a real database). Also spotted an architecture smell — external HTTP calls made *inside* the transaction — flagged for refactor.
- **Wrote a documented, checklist-based study plan** so the remaining topics (security, caching, core Java, concurrency) are structured and tracked.

### What I'm planning
- Continue the backend plan — Spring Security (server-side), caching, core-Java/concurrency refreshers — using the project as the practice lab.
- Replicate the test suite across the other services.

### Problems & solutions
- **The framework's brand-new major version had moved things around.** The usual test helpers and the database-container integration weren't where the docs expect (bleeding-edge version); I found working equivalents and **noted them** so the next service goes faster.

### Key takeaways
- **Tests force real understanding.** Writing them sent me back deep into the Java/Spring code — exactly the muscle a technical test exercises. *Measuring* a problem (query counts) and *proving* a fix beats reciting theory.
- **Know the classic traps cold.** N+1 and transaction rollback come up in almost every backend interview; having demonstrated both on real code, I can explain them with confidence.
